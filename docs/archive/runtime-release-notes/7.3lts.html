Databricks Runtime 7.3 LTS (unsupported)

Databricks released this image in September 2020. It was declared Long Term Support (LTS) in October 2020.

The following release notes provide information about Databricks Runtime 7.3 LTS, powered by Apache Spark 3.0.

For help with migration from Databricks Runtime 6.x, see Databricks Runtime 7.x migration guide (unsupported).

New features

Delta Lake features and improvements

This release provides the following Delta Lake features and improvements:

In this section

  • Delta Lake performance optimizations significantly reduce overhead

  • Clone metrics now available

  • Delta Lake MERGE INTO improvements

  • Specify the initial position for Delta Lake Structured Streaming source

  • Multi-cluster writes from spark-submit jobs

Delta Lake performance optimizations significantly reduce overhead

This release enables a collection of optimizations that reduce the overhead of Delta Lake operations from seconds to tens of milliseconds. To enable these optimizations you must upgrade all of your clusters that write to and read your Delta table to Databricks Runtime 7.3 LTS or above.

For details and caveats, see Enhanced checkpoints for low-latency queries.

Clone metrics now available

Clone operation metrics are now recorded and can be viewed when you run DESCRIBE HISTORY. For details, see Review Delta Lake table details with describe detail.

Delta Lake MERGE INTO improvements

  • Merge now supports any number of MATCHED and NOT MATCHED clauses.

  • Merge queries that unconditionally delete matched rows no longer throw errors on multiple matches.

Specify the initial position for Delta Lake Structured Streaming source

The Delta Lake Structured Streaming connector now supports startingVersion and startingTimestamp options to specify the starting point of the streaming query without processing the entire table. For details, see Specify initial position.

Multi-cluster writes from spark-submit jobs

You can now write to a Delta table from concurrent spark-submit jobs running on multiple clusters with full ACID guarantees.

Upgraded S3 connector is generally available

The upgraded S3 connector is based on Hadoop 3.1.3 and contains many stability improvements (see Hadoop stability epics 2, 3 and 4), including optimizations on LIST that can lead to significant LIST cost reduction, depending on depth and breath of the S3 directory structure. For details, see Connect to Amazon S3.

Adaptive query execution

Adaptive query execution (AQE) is a query re-optimization framework that dynamically adjusts query plans during execution based on runtime statistics collected. AQE is enabled by default in Databricks Runtime 7.3 LTS. You can now try out all AQE features. For details, see Adaptive query execution.

Improvements

Auto Loader

  • Now supports Azure Data Lake Storage Gen1 in directory listing mode

  • If the file format is text or binaryFile you no longer need to provide the schema. Because the two file formats have a fixed schema, Auto Loader can automatically use a fixed schema.

New Azure Synapse Analytics connector column length control

You can now use a new parameter maxbinlength to control the column length of BinaryType columns. This parameter is translated as VARBINARY(maxbinlength).

The previous (un-configurable) default was VARBINARY(MAX), which caused issues when creating tables with binary columns and the default indexing scheme. Specifically, the table creation would fail, as the default columnstore index does not support MAX in the string and binary columns.

Behavior of dbutils.credentials.showRoles

  • dbutils.credentials.showRoles() now shows your current default role, if you have one, rather than showing an empty list. For passthrough with SAML 2.0 federation, your default role is the first role of all your roles in alphabetical order. For passthrough with Databricks SCIM, you have a default role only if you have one role in total.

  • RuntimeException: Credential Passthrough is now thrown when users try to use dbutils.credentials without credential passthrough enabled.

Kinesis starting position for stream using at_timestamp

The Kinesis connector now supports a starting position for a stream using the at_timestamp option instead of just starting from the earliest or latest values. The configuration is passed as a JSON string to the initialPosition option, for example: "{"at_timestamp": "06/25/2020 10:23:45 PDT", format: "MM/dd/yyyy HH:mm:ss ZZZ"}". See Connect to Amazon Kinesis.

pandas to Spark DataFrame conversion simplified

To enable the following improvements, set the Spark configuration spark.databricks.execution.pandasZeroConfConversion.enabled:

  • createDataFrame(pandas_df) has been optimized on Databricks.

  • A warning is displayed if the conversion cannot be done efficiently or is not possible.

New maxResultSize in toPandas() call

A new configuration setting (spark.databricks.driver.localMaxResultSize) lets you control maxResultSize for a df.toPandas() call. The limit is temporarily changed to max(spark.driver.maxResultSize, spark.databricks.driver.localMaxResultSize). The default value for the configuration setting is the physical memory size divided by 4.

A warning is displayed if the df.toPandas() result size is greater than spark.driver.maxResultSize and less than spark.databricks.driver.localMaxResultSize.

This configuration setting affects only DataFrames created by a df.toPandas() call.

Note

spark.databricks.driver.localMaxResultSize takes effect only when spark.databricks.execution.pandasZeroConfConversion.enabled is true.

Debuggability of pandas and PySpark UDFs

When a pandas or PySpark UDF fails in a Databricks notebook, the error message now includes the line number where the code failed and information about the root cause.

Library upgrades

  • Upgraded Python libraries:

    • koalas upgraded from 1.1.0 to 1.2.0.

    • pyarrow upgraded from 0.15.1 to 1.0.1.

  • Upgraded several installed R libraries. See Installed R Libraries.

  • OpenJDK 8 build is changed from Ubuntu OpenJDK 8 to Zulu OpenJDK 8.
