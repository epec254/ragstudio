Databricks Runtime 5.0 (unsupported)

Databricks released this image in November 2018.

The following release notes provide information about Databricks Runtime 5.0, powered by Apache Spark.

New Features

  • Delta Lake

    • Subqueries are now supported in the WHERE clause for the support for DELETE and UPDATE commands.

    • New scalable implementation for MERGE commands.

      • No limit on the number of inserts and updates.

      • Can be used for SCD Type 1 and Type 2 queries.

      • Can be to perform upserts from streaming queries in “update” mode (for example, write streaming aggregation output to a Delta table). See example Writing streaming aggregates into Databricks Delta Using MERGE and foreachBatch notebook.

  • Structured Streaming

    • Azure Blob storage file notification based streaming source. This can significantly reduce listing costs when running a Structured Streaming query on files in Azure Blob storage. Instead of using listing to find new files for processing, this streaming source can directly read file event notifications to find new files. See Azure Blob storage file source with Azure Queue Storage (legacy).

Improvements

  • Delta Lake

    • OPTIMIZE performance and stability.

      • The OPTIMIZE command commits batches as soon as possible, instead of at the end.

      • Reduced the default number of threads OPTIMIZE runs in parallel. This is a strict performance increase for large tables.

      • Sped up OPTIMIZE writes by avoiding an unnecessarily sorting data when writing to a partitioned table.

      • Sped up OPTIMIZE ZORDER BY by making it incremental. This means the command now avoids rewriting data files that were already Z-ordered by the same column(s). See Data skipping with Z-order indexes for Delta Lake.

    • Snapshot isolation when querying Delta tables. Any query with multiple references to a Delta table (for example, self-join) reads from the same table snapshot even if there are concurrent updates to the table.

    • Improved query latency when reading from small (< 2000 files) Delta tables by caching metadata on the driver.

  • Improved MLlib logistic regression performance.

  • Improved MLlib tree algorithm performance.

  • Upgraded several Java and Scala libraries. See Installed Java and Scala libraries (Scala 2.11 cluster version).

  • Upgraded some installed Python libraries:

    • pip: 10.0.1 to 18.0

    • setuptools: 39.2.0 to 40.4.1

    • tornado: 5.0.2 to 5.1.1

  • Upgraded several installed R libraries. See Installed R Libraries.

Bug Fixes

  • Delta Lake

    • Configurations set in SQL conf now correctly apply to Delta Lake operations that were first loaded in a different notebook.

    • Fixed a bug in DELETE command that would incorrectly delete the rows where the condition evaluates to null.

    • Streams that take more than two days to process the initial batch (that is, data that was in the table when the stream started) no longer fail with FileNotFoundException when attempting to recover from a checkpoint.

    • Avoids a race condition that lead to NoClassDefError when loading a new table.

    • Fix for VACUUM where the operation may fail with an AssertionError stating: “Shouldn’t have any absolute paths for deletion here.”

    • Fixed SHOW CREATE TABLE command to not include Hive-generated storage properties.

  • Executors that throw many NoClassDefFoundError errors for internal Spark classes are now automatically restarted to fix the issue.

Known Issues

  • Column names specified in the replaceWhere option for overwrite mode in Delta Lake are case sensitive even if case insensitivity is enabled (which is the default).

  • The Snowflake connector for Databricks Runtime 5.0 is in Preview.

  • If you cancel a running streaming cell in a notebook attached to a Databricks Runtime 5.0 cluster, you cannot run any subsequent commands in the notebook unless you clear notebook state or restart the cluster. For a workaround, see the Knowledge Base.
