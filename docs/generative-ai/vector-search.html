Databricks Vector Search

Preview

This feature is in Public Preview.

This article gives an overview of Databricks’ vector database solution, Databricks Vector Search, including what it is and how it works.

What is Databricks Vector Search?

Databricks Vector Search is serverless similarity search engine that allows you to store a vector representation of your data, including metadata, in a vector database. With Vector Search, you can create auto-updating vector search indexes from Delta tables managed by Unity Catalog and query them with a simple API to return the most similar vectors.

Vector Search integrates with the following Databricks features:

  • Delta tables contain the input data for vector databases. With Vector Search, you can create a vector search index from a Delta table, and automatically sync that index with contents of a Delta table.

  • Unity Catalog handles data governance and access control lists. You can manage access to both the Vector Search API and the underlying databases with Unity Catalog.

  • Serverless compute. Vector Search runs on serverless compute, which means that servers and infrastructure management is handled in your Databricks account.

  • Model Serving automates querying of the model serving endpoint for embedding generation.

You can apply Vector Search in these use cases:

  • Retrieval augment generation (RAG) systems: Vector Search allows for efficient retrieval and augmentation of data. This leads to more accurate responses and minimizes errors, such as hallucinations in LLM outputs.

  • Recommendation systems: Utilized by e-commerce and streaming platforms for efficient nearest-neighbor searches, matching user behavior with relevant items.

  • Image and video recognition: Facilitates quick searches for similar features in images and videos.

  • Bioinformatics: Applies to tasks like DNA sequence alignment and protein structure similarity search.

  • Anomaly detection: Identifies outliers in contexts like fraud detection or network security.

How does Vector Search work?

Vector Search works differently based on how you want to handle computing embedding vectors and manage your vector index.

At a high-level, you can work with Vector Search as follows:

  1. You create a vector search endpoint. This endpoint is used to create and access vector search indexes. You can create vector search endpoints with the REST API or the Compute UI. Each endpoint scales automatically with the number of vectors in the index and the number of concurrent requests.

  2. You create a vector search index. This index can be based on a specified column of a Delta table that contains your chunked data. See Vector search index for details on the supported index types.

  3. Vector Search ingests the data chunks as inputs and maps them to an N-dimensional space represented by a sequence of numbers known as an embedding vector.

  4. Vector Search uses the embedding vector with a property so that similar concepts are mapped closer together than dissimilar concepts. For example, apples and oranges are mapped closer to each other than apples and boats.

  5. Vector Search stores these vectors in a vector database that provides fast and scalable nearest-neighbor search algorithms to find similar vectors.

    1. Vector Search uses Hierarchical Navigable Small World (HNSW) for its approximate nearest neighbor searches and the cosine similarity distance metric to measure embedding vector similarity.

  6. You query the vector database using your vector search endpoint.

Requirements

  • Unity Catalog enabled workspace.

  • Serverless compute enabled.

  • Source table must have Change Data Feed enabled.

Vector search index

A vector search index is a representation of your embedding vectors that supports real-time and efficient approximate nearest neighbor search queries.

In Vector Search, the index for embeddings is generated from a specific column in a source Delta table. A vector search index is served by a vector search endpoint that you can query and update using the REST API or SDK. A vector search endpoint can contain multiple indexes.

Vector search indexes appear as table-like objects within Unity Catalog. Access to vector indexes and index-level permissions such as querying, updating and creation are governed by Unity Catalog.

Vector Search supports two index types:

  • Delta Sync Index automatically syncs with a source Delta table and incrementally updates as the underlying data in the Delta table changes. You can choose to have Databricks manage embedding vectors for you or manage them yourself.

  • Direct Vector Access Index supports direct read and write of embedding vectors and metadata through a REST API or an SDK. For this index, you manage embedding vectors and index updates yourself.

The following sections provide a walkthrough of how Vector Search works with each of these index scenarios.

Delta Sync Index with managed embeddings

For automatic embeddings generation and Databricks-managed embeddings, create a Delta Sync Index.

In this scenario,

  1. You provide a source Delta table with your chunked data.

  2. You provide a model serving endpoint that serves your preferred embedding model. You can choose any embedding model that suits your needs, like an open source model from OpenAI or a custom built model you trained.

  3. Vector Search uses this embedding model to generate the embeddings from the data in the Delta table.

  4. As the source data in the Delta table is updated, Vector Search updates the vector index as long as Change Data Feed is enabled on the source Delta table.

  5. Two update modes are supported: triggered and continuous. The continuous mode keeps the index in sync with seconds of latency. However, it has a higher cost associated with it since a compute cluster is provisioned to run the continuous sync streaming pipeline.

  6. You can query this index and vector database using the Delta Sync API.

Delta Sync Index with self-managed embeddings

You can also create a Delta Sync Index and compute the embeddings yourself. This option is applicable for customers who have already pre-generated embeddings or require external access, like for transfer learning or re-ranking to embeddings.

In this scenario,

  1. You provide your embedding vectors in a Delta table.

  2. Vector Search creates the index based on the column you specify from the source Delta table.

  3. As the source data in the Delta table is updated, Vector Search updates the vector index as long as Change Data Feed is enabled on the source Delta table.

  4. Two update modes are supported: triggered and continuous. The continuous mode keeps the index in sync with seconds of latency. However, it has a higher cost associated with it since a compute cluster is provisioned to run the continuous sync streaming pipeline.

  5. You can query this index and vector database using the Delta Sync API.

Direct Vector Access Index

If you have existing pipelines outside of Databricks or are already using a vector database, you can create a Direct Vector Access Index.

In this scenario,

  1. You provide your embedding vectors in a Delta table.

  2. Vector Search creates the index based on the column you specify from the source Delta table.

  3. The index still automatically scales and provides real-time performance, but it’s up to the user to manage updates to the index from source data.

  4. You can query this index and vector database using the Direct Vector Access API.

Use Vector Search

You can create and manage the Vector Search components, like a vector search endpoint and vector search index, using the UI, Python SDK, and REST API. Databricks recommends using the UI for the simplest experience, see How to create and query a Vector Search index.

Limitations

  • Vector Search is not supported for workspaces that use PrivateLink or IP access lists.

  • Customer Managed Keys (CMK) are not supported for the Public Preview.

  • Regulated workspaces are not supported, therefore this functionality is not HIPAA compliant.

  • Row and column level permissions are not supported. However, many customers implement their own application level ACLs using the filter API.

  • This functionality is only available in the following regions:

    • ap-southeast-2

    • eu-west-1

    • us-east-1

    • us-east-2

    • us-west-2
