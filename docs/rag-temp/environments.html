Environments

Context: RAG development workflows

The workflow for developing RAG Applications requires that you let your end users interact with early versions of the application to provide feedback. Without this feedback, it is imposssible to iterate on and improve the application‚Äôs quality. Unlike traditional machine learning projects, allow users to interact with early versions of a RAG Application requires you to deploy infrastructure, such as the Review UI as a web-based chat-app and the corresponding REST API endpoint that supports the Review UI, and allow that infrastructure to access your production data.

However, even though the need for infrastructure to support collecting user feedback & the requirement to use production data access during development testing can create a blurry line between development and production environments, it is critically important that developers maintain a clean seperation between these environments.

Beyond the above requirements, it is ideal if the development version of the application - once successfully validated through user feedback - can be quickly deployed to the production environment using the same code base (e.g., no code changes, only configuration changes). This is the standard paradigm in full stack application development. Without the ability to have a shared code base between development and production, developers are forced to maintain and sync seperate code bases - leading to the risk that the application exhibits online/offline skew, and, in the worst case, fails to deliver high quality answers in production.

How Environments work in RAG Studio

To account for this workflow and accelerate your time to market, RAG Studio provides first-class support for Environments, which enable you to have a single code base that is ‚Äúby-default‚Äù deployable via CI/CD with only configuration changes between dev and prod. RAG Studio‚Äôs Environments are based on Databricks Asset Bundles concept of targets.

To understand how Environments work, let‚Äôs review the components of a RAG Studio application.

Note

üöß Roadmap üöß

RAG Studio only supports a single Workspace for all environments. Support for RAG Studio applications that use multiple Workspaces is on our roadmap.

Each RAG Application comes with 3 pre-configured environments:

  1. End Users Environment end_users Used for production traffic

  2. Reviewers Environment reviewers Used for testing versions with end users/expert reviewers

  3. Development Environment dev Used for the inner development loop e.g., testing and rapid iteration

Optionally, to support multiple developers working on the same application, you can configure more than 1 dev environment.

Note

üöß Roadmap üöß

RAG Studio development currently requires access to a Databricks Workspace for compute. Support for a fully offline development environment using only local compute is on our roadmap.
